# 家計簿アプリ Web化リファクタリング設計書

作成日: 2026-07-18
ステータス: 承認済み（実装開始）

## 背景と目的

現行の家計簿アプリは Google Apps Script (GAS) + スプレッドシート上で動作しているが、
カスタマイズ性に限界があるため、Web アプリケーションへ全面リファクタリングする。

### 必須要件

1. **アクセス制限**: 夫婦2人以外は絶対にアクセスできないこと
2. **モバイルUX**: スマートフォンから見やすく・操作しやすい UI/UX（妻の主要利用環境）
3. **機能維持**: 現行実装されている全機能がそのまま使えること
4. **CSV一括インポート**: 複数のCSVファイルを同時にインポートできること（夫のPC利用を想定）

## 全体アーキテクチャ

```
[スマホ/PC ブラウザ]
      │ HTTPS
      ▼
[Cloudflare Access]  ← Google ログイン + メールアドレス2件のみ許可（第1の壁）
      │
      ▼
[Cloudflare Workers]
  ├─ React SPA (Vite + Tailwind) … モバイルファーストUI、PWA対応
  └─ API (Hono) … Access JWT検証（第2の壁）+ ビジネスロジック
      │
      ▼
[Cloudflare D1 (SQLite)] … 取引・設定・証券残高のDB（外部非公開）
```

- すべて Cloudflare 無料枠で運用（月額0円・サーバー管理不要）
- 技術スタック: TypeScript + React (Vite) + Hono + D1、テストは Vitest

### セキュリティ設計（二重の防御）

1. **Cloudflare Access (Zero Trust)**: アプリの手前で Google ログインを強制し、
   許可メールアドレスを夫婦の2件に限定。未認証リクエストはアプリに到達しない。
2. **API層での JWT 再検証**: Access が発行する `Cf-Access-Jwt-Assertion` を
   毎リクエスト検証し、メールアドレスを再チェック（Access 設定ミスへの保険）。
3. D1 は Workers からのみ到達可能。インターネットに露出しない。
4. PWA としてホーム画面に追加可能。Google アカウントでのログインのため
   新規パスワード管理は不要。

## データベース設計

```sql
transactions        (id, date, description, amount, type, institution,
                     category, memo, balance, import_hash)
                     -- import_hash: 重複取込防止用ハッシュ（現行にない改善）
securities_balances (id, date, brokerage, value)
category_rules      (id, keyword, institution, category, priority)
                     -- 「イオンカード×十日市場→食料品」のハードコード特例を
                     -- 金融機関+キーワードの複合条件ルールとして一般化
csv_formats         (id, name, date_col, desc_col, expense_col,
                     income_col, balance_col, header_rows, encoding)
split_rules         (id, match_type[keyword|institution], pattern, rate)
                     -- 50 / 31 / 100 の負担率を統一表現。
                     -- 「ﾖｺﾊﾏｼﾎｲｸﾘﾖｳ→31%」のハードコードも設定データ化
excluded_categories (id, category, scope[balance|annual])
```

- カテゴリ `振替` は現行同様、収支計算から常に除外（アプリロジック側で保証）
- 金額・残高は整数（円）で保持

| 割り勘計算（50% / 保育料31% / 全額100%） | 割り勘画面（負担率は設定テーブル化、既定値に現行ルール投入） |
| 証券残高の手入力（シート直接編集） | 証券残高入力フォーム |

### 画面構成

- **ダッシュボード**（ホーム）: 今月の収支・カテゴリ別支出円グラフ・総資産を1画面で
- **月次レポート**: 年月切替、カテゴリタップで明細ドリルダウン
- **年間レポート**: 直近12ヶ月の収支/投資推移とカテゴリ別集計
- **資産**: 総資産推移グラフ + ポートフォリオ円グラフ + 証券残高入力
- **割り勘計算**: 当月請求額をワンタップ表示（最短導線）
- **明細一覧**: 検索・フィルタ・その場でカテゴリ修正
- **CSVインポート**: 複数ファイル一括取込
- **設定**: 分類ルール / CSVフォーマット / 割り勘ルール / 除外カテゴリの編集

グラフは Recharts、UI は Tailwind CSS を使用。モバイルファースト
（下部タブナビゲーション）だが PC でも快適に使える responsive 設計。

## CSVインポート機能（複数ファイル一括）

### UIフロー

1. **複数ファイルを一括受付**: PC はドラッグ&ドロップ、スマホは `multiple` ファイル選択
2. ファイルごとに**フォーマット自動判定**（ヘッダー行内容・列構成・エンコーディングの照合）
   し、パース結果をプレビュー（件数・期間・重複件数）。判定失敗分のみ手動選択
3. **「まとめてインポート」1ボタンで全ファイル処理**。ファイルごとに
   成功件数／重複スキップ件数／エラーを結果一覧表示

### 設計ポイント

- **部分失敗の許容**: ファイルごとに独立トランザクション。1ファイルのエラーが
  他に波及しない
- **重複防止**: `import_hash`（日付+内容+金額+種別+金融機関のハッシュ）により
  同一CSVの再アップロードは自動スキップ+件数報告
- **自動判定の学習**: 手動選択時にヘッダー特徴・ファイル名パターンを記録し、
  次回から自動判定

## 移行ステップ

1. **足場作り**: webapp/ の雛形（Vite + React + Hono + Workers + D1）、
   スキーマ・マイグレーション定義
2. **データ移行**: 現スプレッドシートの全シートを CSV エクスポート → D1 投入する
   移行スクリプト
3. **API移植**: CsvImporter / Categorizer / ReportGenerator / SplitwiseCalculator を
   テスト付きで移植。既存データで現行版と計算結果を突き合わせ検証
4. **UI実装**: モバイルファーストで各画面を実装、PWA化
5. **認証・デプロイ**: Cloudflare Access 設定（メール2件許可）、デプロイ
6. **並行運用**: 1〜2ヶ月 GAS 版と併用しレポート数値の一致を確認後、GAS 版停止
