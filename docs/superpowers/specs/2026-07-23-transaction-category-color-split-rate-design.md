# 明細カテゴリ色・割り勘割合表示 設計

## 背景

明細画面ではカテゴリを一律の青緑色で表示しており、割り勘対象かどうかは表示していない。

月次レポートの円グラフと内訳は、支出額の多い順に並んだカテゴリへカラーパレットを先頭から割り当てている。このため、月や集計結果が変わってカテゴリの順位が入れ替わると、同じカテゴリでも色が変わる。ホームと年間レポートのカテゴリ円グラフも同じ並び順依存の方式を利用している。

割り勘処理には、複数ルールへ一致した場合に `priority` の小さいルールを優先する仕組みが導入済みである。同順位の場合は、金融機関ルール、キーワードルール、割合の降順、IDの昇順で評価する。

## 目標

- 明細、ホーム、月次レポート、年間レポートで、同じカテゴリを常に同じ色で表示する。
- カテゴリの色は設定項目を増やさず、カテゴリ名から自動的かつ決定的に求める。
- 割り勘対象の明細カードに、最優先ルールから決まった相手負担割合を表示する。
- 割り勘レポートと明細カードで、対象判定と適用割合が食い違わないようにする。
- 既存データを変更せず、割り勘ルールの変更を次回表示から反映する。

## 対象外

- カテゴリ色を利用者が編集する設定
- 割り勘対象や割合による明細フィルタ
- 明細画面からの割り勘ルール編集
- 割り勘判定結果のDB保存
- 資産ポートフォリオなど、カテゴリ以外のグラフの配色変更
- 割り勘ルールの割合・優先度に対する入力制約の変更

## 採用方式

カテゴリ色は、クライアントの共通関数でカテゴリ名から決定する。DBには色を保存しない。

割り勘情報は、明細APIが既存の共有判定ロジックを利用して算出し、一覧表示用の派生項目として返す。ブラウザでルールを別途取得して判定する方式や、判定結果を取引へ保存する方式は採用しない。

この方式には次の利点がある。

- カテゴリの表示順や月に依存せず、すべてのカテゴリ表示で同じ色を利用できる。
- 新しいカテゴリに設定作業なしで色が付く。
- 割り勘の優先順位を一か所で管理できる。
- ルール変更後の再計算やデータ移行が不要になる。

## カテゴリ色

### 色決定

クライアントにカテゴリ色専用のモジュールを設け、現在のチャート用カラーパレットと `getCategoryColor` を配置する。

`getCategoryColor` は次の順に色を決める。

1. カテゴリ名の前後空白を除去する。
2. 空文字またはNULLは `未分類` として扱う。
3. `未分類` には固定の中立色を返す。
4. それ以外はカテゴリ名をUTF-8のバイト列に変換し、32ビットFNV-1aハッシュを適用する。
5. 符号なしハッシュ値を固定パレットの要素数へ対応させ、色を返す。

同じ入力に対する色を将来も維持するため、ハッシュ方式、パレットの内容、パレットの順序を表示上の互換仕様として固定する。これらを変更する場合は、既存カテゴリ色が変わる意図的な仕様変更として扱う。

有限のパレットを自動割当するため、異なるカテゴリが同じ色になる可能性は許容する。すべての対象画面でカテゴリ名を併記し、色だけを識別手段にしない。

### グラフ

汎用の `CategoryPie` は、名前から色を返す任意のコールバックを受け取れるようにする。

- ホームのカテゴリ別支出
- 月次レポートのカテゴリ別支出
- 年間レポートの月平均支出

上記は `getCategoryColor` を渡す。月次レポートの内訳にある色見本も同じ関数を使う。支出額による表示順は変更せず、色の決定だけをカテゴリ名基準へ変える。

カテゴリを扱わない資産ポートフォリオは色決定コールバックを渡さず、現在の並び順による配色を維持する。

### 明細カード

カテゴリ表示は、共通関数で得た色を使ったドット、薄い背景、カテゴリ名で構成する。文字色は十分なコントラストを持つ中立色とし、カテゴリ色そのものを文字の判読性に依存させない。

NULLまたは空のカテゴリは `未分類` と表示し、固定の中立色を使う。

## 割り勘判定

### 共有判定

共有の割り勘モジュールに、取引の対象条件とルール一致をまとめた関数を追加する。この関数は、評価順へ並べ替え済みのルールと取引を受け取り、適用する `SplitRule` または `null` を返す。

対象条件は既存の割り勘レポートと同じく、次の両方を満たすことである。

- 種別が `支出`
- カテゴリが `振替` ではない

対象条件を満たした取引は、既存の部分一致と優先順位で最初に一致したルールを採用する。

ルールの評価順は既存の `sortSplitRules` だけで決める。

1. `priority` の昇順
2. 同順位では金融機関ルールをキーワードルールより先にする
3. 同じ種別では `rate` の降順
4. 残る同順位はIDの昇順

リポジトリのSQL取得順には依存しない。各利用箇所はルール一式を一度だけ `sortSplitRules` へ渡し、並べ替えた結果を複数明細の判定で再利用する。

既存の `calculateSplitwise` も新しい共有判定関数を使う。これにより、明細APIと割り勘レポートで対象条件、優先順位、部分一致の仕様を共有する。

### 明細API

`GET /api/transactions` は、ページ対象の明細と割り勘ルールを並行して取得する。ルールを一度だけ並べ替え、ページ内の各明細に共有判定を適用する。

保存済み取引を表す `Transaction` 型は変更しない。一覧レスポンスには次の派生型を用いる。

```ts
type TransactionListItem = Transaction & {
  splitRate: number | null;
};
```

一致したルールがある場合は、その `rate` を `splitRate` に設定する。対象条件を満たさない場合、または一致ルールがない場合は `null` にする。優先度、パターン、ルールIDは一覧レスポンスへ含めない。

DBスキーマと保存データは変更しない。割り勘ルールや取引カテゴリを変更した場合は、次回の一覧取得で `splitRate` を再計算する。

## 表示仕様

明細カードの補足情報は、次の順で表示する。

1. 日付
2. 金融機関
3. カテゴリ
4. 相手負担割合
5. メモ

`splitRate` が数値の場合だけ、カテゴリとは区別できる色のバッジで `相手負担 50%`、`相手負担 100%` などと表示する。`splitRate` が `null` の場合は、バッジも「対象外」という文言も表示しない。

補足情報は狭い画面で折り返す。割合は文字として明示し、バッジの色だけに意味を持たせない。編集モーダルの項目と、カードを押して編集する既存操作は変更しない。

## エラー処理

明細または割り勘ルールの取得・判定に失敗した場合は、誤って対象外に見せないため、明細一覧APIをエラーにする。クライアントは既存のエラー表示と再試行導線を利用する。割り勘情報だけを黙って省略する縮退表示は行わない。

カテゴリ色はクライアント内の純粋関数で決定するため、通信エラーは発生しない。

## テスト

### カテゴリ色

- 同じカテゴリ名が常に同じ色になる。
- 表示順や金額順位が色へ影響しない。
- 前後空白を除いた同じ名前が同じ色になる。
- NULL、空文字、`未分類` が固定の中立色になる。
- 代表カテゴリと色の組み合わせを固定値で検証し、ハッシュやパレットの意図しない変更を検出する。

### 割り勘

- キーワードと金融機関の部分一致を維持する。
- `priority` の小さい一致ルールが選ばれる。
- 同順位の既存タイブレーク順を維持する。
- 収入と `振替` は対象外になる。
- 50%、100%、31%など、選ばれたルールの割合をそのまま返す。
- `calculateSplitwise` の対象明細、割合、合計額が変わらない。

### 明細API

- 対象明細に最優先ルールの `rate` が `splitRate` として付く。
- 複数ルールへ一致する場合も、割り勘レポートと同じルールの割合になる。
- 対象外または不一致の明細に `splitRate: null` が付く。
- ページング、月、カテゴリ、金融機関、キーワードの既存絞り込み結果が変わらない。

## 実装後の検証

- `npm test` で既存テストと追加テストを実行する。
- `npm run build` で型検査と本番ビルドを実行する。
- 月を切り替えても同じカテゴリ色が維持されることを、ホーム、月次レポート、明細で確認する。
- 年間レポートでも同じカテゴリ色が使われることを確認する。
- 50%、100%、対象外の各明細表示を確認する。
- 複数ルールへ一致する明細で、割り勘レポートと明細カードが同じ割合を表示することを確認する。
- 狭い画面でカテゴリと相手負担割合が読みやすく折り返されることを確認する。

## 完了条件

- 同じカテゴリが対象画面間および月をまたいで同じ色になる。
- 明細カードに、最優先の割り勘ルールから得た相手負担割合が表示される。
- 割り勘対象外の明細に余分な表示が追加されない。
- 割り勘レポートと明細カードの対象判定・割合が一致する。
- DBマイグレーションとカテゴリ色設定を追加しない。
- 既存テスト、追加テスト、本番ビルドが成功する。
