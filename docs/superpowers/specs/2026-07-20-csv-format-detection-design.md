# CSVフォーマット自動判定 改修設計

## 背景

現行のWeb版は、登録された各CSVフォーマットで取引を試験解析し、日付有効率と取引有効率の重み付きスコアで形式を推測している。この方式では、列位置が同じ金融機関を区別できず、文字コードと候補フォーマットの対応も保証されない。また、僅かなスコア差で誤った候補を採用する余地がある。

gitignore対象の `input/csv_format` にある9金融機関のサンプルを調査したところ、ヘッダー付き形式は列名で明確に識別できる。ヘッダーのない2形式は想定列数が異なるため、列構造と既存の列マッピングで区別できる。

## 目標

- 対象の9サンプルを、それぞれ正しい金融機関として自動判定する。
- 未知形式、設定競合、構造変更のあるCSVを推測で取り込まず、手動選択へ回す。
- 新しい金融機関をコード変更なしで追加できるよう、判定条件を `csv_formats` で管理する。
- 実際の取引内容、氏名、番号、金額などを、追跡対象ファイル、テスト、ログへ含めない。

## 対象外

- 区切り文字がカンマ以外のファイルへの対応
- CSV設定をサンプルファイルから自動生成する機能
- ファイル名による金融機関判定
- 旧GAS版の自動インポート処理の変更

## 設定モデル

`csv_formats` に次の列を追加する。

| 列 | 型 | 内容 |
|---|---|---|
| `header_signature` | TEXT NULL | 識別に使うヘッダー1行をCSV形式で保持する。ヘッダーなし形式はNULL |
| `expected_columns` | INTEGER NULL | データ行の想定列数。設定する場合は1以上 |

`header_signature` は、スキップ対象となる先頭 `header_rows` 行のいずれかと比較する。識別用ヘッダーの後に個人別の前置き行を持つ形式にも対応するためである。したがって、ヘッダー署名を持つフォーマットは `header_rows >= 1` を必須とする。

ヘッダー比較では、CSVとしてセルへ分割した後、各セルに次の正規化を適用する。

1. 先頭セルのBOMを除去する。
2. Unicode NFKC正規化を行う。
3. 前後空白を除去する。
4. 連続する空白文字を単一の半角スペースへ統一する。

列順とセル数は厳密に比較する。月など可変のヘッダーセルに限り、署名側の `*` をワイルドカードとして扱う。金融機関側で列追加や並べ替えが行われた場合は、既存マッピングで取り込まず判定不能にする。

9金融機関の既存設定は、新しいマイグレーションでヘッダー署名と想定列数を補完する。三井住友カードと東急カードはヘッダー署名をNULLにする。東急カードのサンプルにはヘッダーがないため、既存の `header_rows` を1から0へ修正する。

利用者が独自に追加した既存フォーマットの想定列数は、安全に推測できないためNULLのまま維持する。判定設定が未完了のフォーマットは自動判定候補から除外し、設定画面で補完を求める。新規作成と更新では1以上の想定列数を必須にする。

## 自動判定アルゴリズム

### 候補評価

各フォーマットは、必ずそのフォーマット自身に設定された文字コードでデコードする。同じ文字コードのデコード結果は内部で再利用してよいが、異なる文字コードで得たテキストを候補間で流用しない。自動判定では不正バイトを置換せず、厳格なデコードに失敗した候補を除外する。

デコードできた候補について、空行を除いたデータ行を対象に以下を計算する。

- ヘッダー一致: `header_signature` がある場合、正規化後の全セルが一致すること。
- 列数一致率: `expected_columns` と列数が一致するデータ行の割合。
- 日付有効率: 設定された日付列を正規化できるデータ行の割合。
- 取引有効率: 日付が有効で、設定された収入列または支出列から非ゼロ金額を解析できるデータ行の割合。
- 解析件数: 実際に生成できた取引件数。

次をすべて満たしたものだけを候補とする。

- ヘッダー署名が設定されている場合は、ヘッダーが一致する。
- 列数一致率が80%以上。
- 日付有効率が60%以上。
- 取引有効率が60%以上。
- 解析件数が1件以上。

### 候補決定

候補は次の優先順位で決定する。

1. ヘッダー署名が一致した候補が1件なら採用する。
2. ヘッダー署名が一致した候補が複数なら、設定競合として判定不能にする。
3. ヘッダー署名の一致候補がなく、ヘッダー署名を持たない候補が1件なら採用する。
4. ヘッダー署名を持たない候補が複数、または候補が0件なら判定不能にする。

重み付きスコアによる強制的な順位付けは行わない。

## インポート処理

自動判定不能の場合はファイルを解析・登録せず、手動選択を要求する。プレビューと本取込は同じ判定処理を利用し、結果が食い違わないようにする。

手動指定時は自動判定を省略し、指定フォーマットの文字コードと列設定で解析する。ただし解析件数が0件の場合は成功扱いにせず、選択した形式では有効な取引を読み取れない旨を返す。

複数ファイルの処理分離、カテゴリ分類、インポートハッシュ、重複排除の既存仕様は変更しない。

## APIと画面

共有の `CsvFormat` 型、設定リポジトリ、設定APIの入力検証へ新しい2項目を追加する。

設定画面では、ヘッダー署名と想定列数を作成・編集できるようにする。保存時に次を検証する。

- `expected_columns` が1以上の整数である。
- `header_signature` がある場合、CSVとして1行だけに解析できる。
- `header_signature` がある場合、`header_rows` が1以上である。

インポート画面では、自動判定が成功した場合も含め、ファイルごとの手動フォーマット選択を常に表示する。選択を変更したらプレビューを無効化し、再プレビューを必要とする。判定不能のファイルは、手動指定して有効な再プレビューが完了するまで取り込めない。

## エラーとプライバシー

内部の候補評価結果には、次のような理由コードを持たせる。

- デコード失敗
- 判定設定不足
- ヘッダー不一致
- 列数一致率不足
- 日付有効率不足
- 取引有効率不足
- 解析結果0件
- 候補競合

画面には解決方法が分かる日本語メッセージを返す。ログ、例外、APIレスポンスにCSVのセル値を含めない。診断情報はフォーマット名、割合、件数、理由コードに限定する。

`input/csv_format` 以下は読み取り専用で扱い、変更、コピー、追跡対象への追加を行わない。

## テストと検証

追跡対象の自動テストでは、架空の店名と金額だけを使った最小fixtureを作成し、次を検証する。

- 9形式相当のヘッダー・列構造を正しく判定できる。
- 同じ列配置でも異なるヘッダー署名を区別できる。
- UTF-8とCP932を取り違えない。
- ヘッダーなし2形式を列数で区別できる。
- 未知ヘッダー、列追加、候補競合、空ファイル、0件CSVを判定不能にする。
- 手動指定で有効な形式を解析でき、誤った形式や0件解析をエラーにする。
- 設定APIが新項目を保存・更新し、不正値を拒否する。

実装後、gitignore内の実サンプル9件に対するローカル検証を実施する。この検証は内容や結果ファイルを保存せず、金融機関名、判定結果、成功・失敗だけを確認する。

## 完了条件

- 実サンプル9件がすべて期待する金融機関へ一意に判定される。
- 未知・曖昧な形式が自動採用されない。
- 設定画面から新しい判定項目を管理できる。
- 既存のCSV解析、プレビュー、取込、重複排除テストが通る。
- 実明細情報が追跡対象ファイルとログに追加されていない。
- 本作業では `git commit` を実行しない。
